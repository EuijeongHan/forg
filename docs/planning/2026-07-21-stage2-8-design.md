## 관통 주제 — "최고"의 정의를 수치로

경쟁(단순 DART 알림봇·증권사 알림)과의 차별점을 한 줄로:
> **놓침 0에 수렴 + 중요도 판별 + 정정 추적 + 정확한 숫자 + 팀 확인.**

각 Stage는 아래 북극성 지표 중 하나를 움직여야 한다:
1. **누락률**(중요공시 중 미도달 비율) → 0
2. **묻힘률**(중요공시가 잡음에 묻히는 비율) → 중요도·필터로 감소
3. **파악 시간**(알림 열고 핵심 이해까지) → 카드·숫자로 단축
4. **정정 반영 시간**(정정 발생→사용자 인지) → 정정 비교로 단축
5. **팀 확인 보장**(중요공시가 최소 1인 확인되는 비율) → ack·재알림

## Stage 2 — 이벤트 카드 (기반 완료, 본계약 대기)

**완료(플래그 뒤)**: `events/` normalizer·metrics·renderer, `DisclosureEvent`, `ENABLE_EVENT_CARDS`.

**본계약에서 결정할 것**:
- **유형 확장 순서** — 설문 Q16/Q17로 확정. 기본안(실측·도메인): 전환사채 → 유상증자 →
  자기주식 → 감자 → 합병·분할 → BW·교환사채 → 무상증자.
- **renderer를 알림 경로에 연결** — 현 `summarizer.format_typed_disclosure`를
  `events.renderer.render_event`로 교체. **컷오버 전략**: 두 출력을 로그로 병행 비교(shadow)
  후 일치 확인 → 플래그로 전환. 카드 숫자는 이미 정형 필드라 회귀 위험 낮음.
- **희석률 활성화** — 분모(발행주식총수)가 필요. `stockTotqySttus` 연동(가이드 §3.6).
  **주의**: 최신 공시 시점 발행주식수와 정기보고서 기준일 사이 시차 → 카드에 기준일 표기 필수.
  분모 없으면 "계산 불가"(0 금지, 이미 구현).

**하지 않을 것**: 모든 정형 유형 한 번에 구현. 실측으로 우선순위 높은 유형부터.

## Stage 3 — 정정 비교 (기반 완료, 본계약 대기)

**완료(파이프라인 미연결)**: `comparator`(diff+중요도), `correction_service`(감지·규칙 매칭·연결),
`DisclosureRelation`.

**본계약에서 결정할 것**:
- **알림 포맷** — 정정공시 감지 시: `[정정]` 헤더 + `render_changes(major_only 우선)` +
  "추정 연결(confidence=rule)" 표기 + 원본/정정본 양쪽 DART 링크.
- **연결 신뢰도 상향** — 규칙 기반(현재)의 한계: 같은 날 동일 제목 복수 공시. **개선 경로**:
  DART 원문에 적힌 관련 접수번호 파싱(있으면 confidence="exact"). 없으면 rule 유지 + 추정 표기.
- **중요/경미 판정 노출** — major(금액·수량·일정·상대방)만 강조, minor(오타·연락처)는 접기.
- **재알림 대상** — 정정이 major면 원본을 확인한 사용자에게도 재통지(중요 변경 놓침 방지).
  → Stage 6(개인화)의 재알림 정책과 연결.

**하지 않을 것**: 과거 비교(다른 사건 간)와 정정 비교를 같은 로직으로 섞기(계획서 §4.1).

## Stage 4 — 팀 확인(ack)·미확인 재알림 ★설문·인터뷰 의존 최대

**왜 중요**: 창업 사례(팀 전체가 분할 공시 놓침)의 본질. "알림"이 아니라 "확인 보장"이 목표.
현행 코드는 분할 키워드로 **알림은 이미 됨** — 놓친 건 확인 보장이었다.

**설계가 갈리는 축(인터뷰 가이드 §3로 결정)**:
| 축 | 옵션 A | 옵션 B | 옵션 C |
|---|---|---|---|
| ack가 사는 곳 | 텔레그램 **그룹**(봇을 방에 초대) | 1:1 각자 + 집계 화면 | 사내 메신저(연동 필요) |
| 확인 방법 | 인라인 버튼 "✅ 확인" | 읽음 이벤트 | 코멘트 |
| 재알림 | N분 후 미확인 시 재발송 | 담당자 지정 후 미확인 에스컬레이션 | 없음 |

**기술 전제**: 현 봇은 1:1 `chat_id` 구조. 그룹 지원은 별도 작업(chat 타입 분기, 그룹 워치리스트
공유 모델). **인터뷰 전 구현 금지** — 스코프가 데이터로 정해지기 전 최대 과투자 위험 지점.

**최소 실험(인터뷰와 병행 가능)**: 알림 메시지에 "✅ 확인" 버튼 하나 + `SummaryFeedback` 재사용
(누가 확인했는지 기록). 그룹/에스컬레이션은 이후.

## Stage 5 — KIND 보완 (측정 먼저)

**완료**: `scripts/compare_dart_kind.py` DART 측 + KIND 미구현(NotImplementedError).
`scripts/measure_disclosure_volume.py` 실측.

**순서(계획서 §6)**: ① KIND 실응답 형식 확인(브라우저 개발자도구) → fixture 저장 →
parser 단위 테스트 → 구현 → ② 5영업일 DART/KIND 비교 리포트 → ③ 수치로 수집 범위 결정.

**결정 대기**: KIND 고유 공시(공정공시·조회공시답변·자율공시 등) 중 기관 실무자에게 중요한 것.
설문·인터뷰로 수요 확인. **하지 않을 것**: KIND 엔드포인트 추측 구현.

## Stage 6 — 개인화 알림·리캡

**설계(계획서 §7)**: `User.today_keywords` 문자열 → `AlertRule`(JSON 조건) 테이블.
조건: event_types, minimum_dilution_rate, after_hours_only, correction_only, important_change_only.
리캡: 저장된 카드·변화 데이터를 Python으로 집계(공시마다 LLM 재호출 금지) + 18:30 KST cron.

**결정 대기**: 기존 사용자 마이그레이션(규칙 없는 사용자는 현 동작 유지). 대량보유 키워드 추가는
실측상 볼륨 부담 낮음(일 1~2건) — 설문 확인 후.

**연결**: Stage 4 재알림 정책, Stage 3 정정 major 재통지가 여기 규칙 엔진으로 수렴.

## Stage 7 — 검증 레이어 + 평가 하네스 ★포폴 코어

**설계(ADR-010 #21 + 계획서 §8)** — 계층형 cascade:
```
요약 생성 → ① 결정론(숫자·단위·엔티티 존재 + 한국어 숫자 정규화 + 투자의견 금칙)
         → ② 정형 교차검증(요약 숫자 == 정형 필드; foRG 데이터 우위)
         → ③ (필요시) LLM claim-judge (supported/contradicted/not-found, 근거 인용)
   판정 → pass:발송 / warning:카드만 / fail:링크만 / unavailable:검증불가 표기
```
- `SummaryVerification`(verdict·checks) + `SummaryFeedback`(👍/⚠️ 버튼) 모델.
- **데이터 플라이휠**: 판정·피드백 적재 → golden set(30~50건) + 검증기 precision/recall 측정.
- **Ragas(ADR-004)** 도입은 이 단계. 임베딩·리랭커·LangGraph 등 무거운 ML은 **여기서도 보류**
  (재검토 조건: Stage 8 가동 + 검색 baseline). 검증은 결정론+정형 교차만으로 큰 값.

**착수 조건**: Stage 2(이벤트 카드) 가동 후 — 검증할 대상(카드·요약)이 정형화돼 있어야 효율.
**선확보 완료**: P1-2 스냅샷이 이미 검증·golden set 재료를 쌓기 시작.

## Stage 8 — 출처 기반 질의 (툴콜링) ★포폴 코어

**설계(계획서 §10)**: 범용 에이전트 금지 → 사전 정의 tool + native tool loop.
tool: get_company_events, get_upcoming_schedules, get_latest_disclosures, compare_disclosures,
get_event_source. LLM은 결과를 서술만(숫자 재계산 금지, tool이 계산). 모든 주장에 공시 링크.
평가: 툴 선택·파라미터 정확도, 조회 recall, 숫자 정확도, 출처 정확도, faithfulness, 투자의견 포함률.
**개인정보 주의**: 사용자 질문이 LLM에 전달됨 → 처리방침 개정 필요(법률 메모 쟁점 3-4).

**착수 조건**: Stage 7 평가 하네스 존재(질의 답변도 같은 검증). #3(챗봇)이 여기로 진화.
