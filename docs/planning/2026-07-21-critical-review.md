## P0 — 지금 즉시 (설문 게시와 무관하게)

### P0-1. CI 부재 — 테스트가 있는데 자동으로 돌지 않는다 [S]
- **문제**: `tests/` 3종(38케이스)을 만들었지만 PR마다 사람이 돌려야 한다.
- **왜**: 회귀를 잡는 건 테스트의 존재가 아니라 **강제 실행**이다. 지금이 도입 최저비용 시점.
- **방안**: GitHub Actions — PR/push 시 `pip install -r requirements.txt -r requirements-dev.txt` 후 3종 실행. PG 서비스 컨테이너로 alembic 멱등성까지 돌리면 금상첨화(선택).

### P0-2. 자정 경계 공시 영구 누락 엣지 [S]
- **문제**: 폴링이 `bgn_de = 오늘(KST)`만 조회한다. 23:5x에 접수된 공시를 처리하기 전에 날짜가 넘어가면 **다시는 조회되지 않는다**.
- **왜**: "놓치지 않는다"가 제품의 존재 이유다. 감사보고서·악재성 공시가 야간에 몰리는 도메인 특성상 실제 위험.
- **방안**: 자정 후 첫 N분(예: 90분)은 전일도 함께 조회하거나, 상시 `bgn_de=어제`로 2일 창 조회(SeenDisclosure·`Disclosure.rcept_no` unique가 중복을 막아 안전). 후자가 단순해 권장.

### P0-3. 파이프라인 침묵 사망 감지 — 자가 경보 [S]
- **문제**: DART 장애·HTML 구조 변경·키 만료 시 폴링이 조용히 빈손으로 돌아도 아무도 모른다. `fetch_rcept_times` 정규식이 깨져도 야간 경고만 조용히 사라진다.
- **왜**: 운영자가 1인이고 대시보드가 없다. 침묵 실패는 "누락 방지" 제품에서 최악의 실패 모드.
- **방안**: 이미 있는 `send_system_message(TELEGRAM_CHAT_ID, …)` 재사용 — ① 폴링 연속 실패 N회(예: 5회) 시 1회 경보, ② 평일 장중인데 오늘 공시 0건이 M사이클 지속 시 경보, ③ rcept_times 결과 0건이 지속되면 경보. 상태는 모듈 전역 카운터로 충분(멀티 인스턴스 아님).

### P0-4. 사용자 액션 2건 (코드 아님 — 리마인더)
- **키 로테이션**: 과거 채팅에 노출된 ANTHROPIC/DART/Telegram/DB 키 — 아직 로테이션 확인 안 됨.
- **Railway 확인**: 최근 배포 로그에서 `DB 마이그레이션 적용 완료` 확인 + (선택) 서비스 TZ 환경변수. 코드가 KST 명시로 바뀌어 TZ 의존은 제거됐지만 로그 타임스탬프 가독성을 위해 `TZ=Asia/Seoul` 권장.

## P1 — 설문·파일럿과 병행

### P1-1. LLM 비용 가드레일 [S]
- **문제**: 일일 LLM 호출 상한이 없다. I-2로 재호출 낭비는 막았지만, 공시 폭주일(분기말 감사보고서 시즌 등)엔 통제 수단이 없다.
- **방안**: 일일 카운터(프로세스 메모리) + 상한 초과 시 요약 생략하고 "카드/제목만 발송" 폴백 + 시스템 경보. 상한은 env로.

### P1-2. Stage 7 자산 선확보 — 정형 데이터 스냅샷 저장 [M]
- **문제**: 지금 매일 흘러가는 (공시, 정형 응답, 요약)이 **저장되지 않고 버려진다**. `SeenDisclosure.summary`만 남고 원천(typed_data·원문 해시)은 유실.
- **왜**: Stage 7(검증 레이어)·golden set·정정 비교(Stage 3)의 재료다. 지금부터 쌓으면 몇 달 뒤 공짜 데이터셋이 된다. 미래 기능의 비용을 오늘의 JSON 컬럼 하나로 선지불하는 셈.
- **방안**: `Disclosure.raw_typed_data JSON` 컬럼 추가(alembic 0002) — tasks에서 typed 조회 성공 시 저장. 원문은 해시+길이만(전문 저장은 보류).

### P1-3. `/today` 상세 요약 캐시 [S]
- **문제**: 같은 공시 버튼을 두 번 누르면 LLM을 두 번 호출한다(조회 경로엔 Seen 개념이 없음).
- **방안**: `disclosure_cache`에 summary 필드 추가(프로세스 메모리, 재시작 시 소실 허용). 또는 P1-2의 스냅샷과 통합.

### P1-4. 감지 격차 검토 — '대량보유'(5%룰)·임원 소유 보고 [결정 필요]
- **문제**: 기관투자자에게 중요한 지분 공시가 `IMPORTANT_REPORT_TYPES`에 없어 **현재 알림 자체가 안 된다**.
- **왜 바로 추가하지 않나**: 5%룰 보고는 건수가 많아 "알림 과다로 묻힘"이라는 원 페인포인트를 악화시킬 수 있다. 키워드 추가는 한 줄이지만 UX 영향이 크다.
- **방안**: 인터뷰(가이드 §2·§5)에서 수요 확인 후, 추가하되 **워치리스트 기업 한정**이라는 현 구조가 이미 볼륨을 제한함을 감안해 판단. 데이터로 결정: 최근 30일 list.json에서 해당 유형 건수 측정 스크립트 먼저(S).

### P1-5. KIND 5영업일 비교 스크립트 [M]
- 로드맵 Stage 5의 선행 조건. 설문 진행 중에 돌려 두면 결과가 함께 도착한다. 계획서 §6.2의 리포트 형식 그대로.
